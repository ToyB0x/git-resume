# コミットメッセージのルール

**更新日**: 2025/3/21
**確認日**: 2025/3/21

このファイルは、git-resumeプロジェクトにおけるコミットメッセージの作成ルールを定めています。一貫性のあるコミットメッセージは、変更履歴の追跡と理解を容易にし、プロジェクトの品質向上に貢献します。

## コミットメッセージの基本形式

```
<type>(<scope>): <subject>

<body>

<footer>
```

- **type**: 変更の種類（必須）
- **scope**: 変更の範囲（任意）
- **subject**: 変更内容の簡潔な説明（必須）
- **body**: 変更の詳細な説明（任意）
- **footer**: 関連する課題番号や破壊的変更の注意事項（任意）

## タイプ（type）の種類

- **feat**: 新機能の追加
- **fix**: バグ修正
- **docs**: ドキュメントのみの変更
- **style**: コードの意味に影響を与えない変更（フォーマット等）
- **refactor**: バグ修正や機能追加ではないコードの変更
- **perf**: パフォーマンスを向上させるコードの変更
- **test**: テストの追加や修正
- **chore**: ビルドプロセスやツール、設定の変更
- **ci**: CI関連の設定やスクリプトの変更

## スコープ（scope）の例

- **api**: APIサーバー関連
- **cli**: CLIツール関連
- **web**: Webフロントエンド関連
- **models**: データモデル関連
- **services**: サービス層関連
- **utils**: ユーティリティ関連
- **infra**: インフラストラクチャ関連
- **deps**: 依存パッケージ関連

## コミットメッセージの例

### 良い例

```
feat(web): ユーザーダッシュボードの実装

- 活動グラフの表示機能を追加
- リポジトリリストの表示を実装
- フィルタリング機能を追加

Closes #123
```

```
fix(api): GitHub APIのレート制限エラーを修正

- トークンプールの実装
- リトライロジックの追加
- エラーハンドリングの改善
```

```
docs: インストール手順の更新と使用例の追加
```

### 悪い例と改善点

```
バグを修正
```
→ 変更タイプとスコープが不明、説明も不十分

```
feat: いろいろ実装
```
→ 具体的な変更内容が不明確

```
add new feature
```
→ 日本語で記述し、変更タイプを明示する

## プルリクエスト時の注意点

- コミットは論理的な単位で分割する
- 複数の変更を含む場合は、可能な限り複数のコミットに分割する
- プルリクエストのタイトルは、主要なコミットメッセージと一致させる

## ブランチ命名規則

ブランチ名は以下の形式に従ってください：

```
<type>/<issue-number>-<short-description>
```

例：
- `feat/123-user-dashboard`
- `fix/456-api-rate-limit`
- `docs/789-update-installation-guide`
