# ドキュメント種別の定義

**更新日**: 2025/3/23
**確認日**: 2025/3/23

## 目的

- プロジェクトに必要なドキュメントの種別を定義する
- 各ドキュメント種別の目的、対象読者、内容要件を明確にする
- ドキュメント作成者が適切な種別を選択できるようにする
- 読者が必要なドキュメントを容易に見つけられるようにする

## ドキュメント種別

git-resumeプロジェクトでは、以下のドキュメント種別を定義します。

### 1. プロジェクト概要ドキュメント

- **目的**: プロジェクトの目的、背景、概要を説明するドキュメント
- **対象読者**: 全プロジェクトメンバー、新規参加者、外部ステークホルダー
- **必須要素**:
  - プロジェクトの目的と価値提案
  - 主要機能の概要
  - 技術スタックの概要
  - 基本的な使用方法
  - セットアップ手順へのリンク
- **配置場所**:
  - リポジトリのルートディレクトリ（README.md）
- **更新頻度**: 主要な機能追加・変更時、四半期ごとの見直し
- **例**: [README.md](/README.md)

### 2. 開発者向けガイド

- **目的**: 開発環境のセットアップ、開発プロセス、コード規約などを説明するドキュメント
- **対象読者**: 開発者、技術リード
- **必須要素**:
  - 開発環境のセットアップ手順
  - ブランチ戦略とコミットルール
  - コーディング規約
  - テスト戦略
  - デバッグとトラブルシューティング
- **配置場所**: docs/guide/developer/ ディレクトリ
- **更新頻度**: 開発環境や開発プロセスの変更時
- **例**: [docs/guide/developer/onboarding.md](/docs/guide/developer/onboarding.md)

### 3. アーキテクチャドキュメント

- **目的**: システムのアーキテクチャ、コンポーネント間の関係、設計原則を説明するドキュメント
- **対象読者**: 開発者、アーキテクト、技術リード
- **必須要素**:
  - システム構成図
  - コンポーネント間の関係と責任分担
  - データフロー
  - 技術選定の理由と制約
  - 非機能要件（スケーラビリティ、セキュリティなど）
- **配置場所**: docs/guide/developer/architecture/ ディレクトリ
- **更新頻度**: アーキテクチャの変更時
- **例**: [docs/guide/developer/architecture/overview.md](/docs/guide/developer/architecture/overview.md)

### 4. アーキテクチャ決定記録（ADR）

- **目的**: アーキテクチャや技術選定に関する重要な決定とその背景を記録するドキュメント
- **対象読者**: 開発者、アーキテクト、技術リード
- **必須要素**:
  - タイトルと日付
  - 決定の背景
  - 検討した選択肢
  - 決定内容とその理由
  - 影響と結果
- **配置場所**: docs/guide/developer/architecture/adr/ ディレクトリ
- **更新頻度**: 重要な技術決定時に新規作成、決定の変更時に更新
- **例**: [docs/guide/developer/architecture/adr/001-spa-mode/README.md](/docs/guide/developer/architecture/adr/001-spa-mode/README.md)

### 5. データモデルドキュメント

- **目的**: データ構造、モデル間の関係、データフローを説明するドキュメント
- **対象読者**: 開発者、データアナリスト
- **必須要素**:
  - エンティティの定義
  - 属性とデータ型
  - モデル間の関係
  - データフロー
  - スキーマ変更履歴
- **配置場所**: docs/guide/developer/architecture/data-models.md
- **更新頻度**: データモデルの変更時
- **例**: [docs/guide/developer/architecture/data-models.md](/docs/guide/developer/architecture/data-models.md)

### 6. APIドキュメント

- **目的**: API仕様、エンドポイント、リクエスト/レスポンス形式を説明するドキュメント
- **対象読者**: 開発者、API利用者
- **必須要素**:
  - エンドポイント一覧
  - リクエスト形式
  - レスポンス形式
  - 認証方法
  - エラーコードと処理
  - 使用例
- **配置場所**: docs/guide/usage/api-guide.md
- **更新頻度**: API変更時
- **例**: [docs/guide/usage/api-guide.md](/docs/guide/usage/api-guide.md)

### 7. ユーザーガイド

- **目的**: エンドユーザー向けの機能説明と使用方法を提供するドキュメント
- **対象読者**: エンドユーザー、サポート担当者
- **必須要素**:
  - 機能の概要
  - 基本的な使用方法
  - 高度な機能と設定
  - よくある質問と回答
  - トラブルシューティング
- **配置場所**: docs/guide/users/ ディレクトリ
- **更新頻度**: 機能追加・変更時
- **例**: [docs/guide/users/web-guide.md](/docs/guide/users/web-guide.md)

### 8. 機能仕様書

- **目的**: 特定の機能の詳細な仕様、要件、振る舞いを説明するドキュメント
- **対象読者**: 開発者、プロダクトマネージャー、QAエンジニア
- **必須要素**:
  - 機能の目的と背景
  - ユーザーストーリーとユースケース
  - 詳細な機能要件
  - UI/UX仕様
  - 技術的な実装詳細
  - テスト条件
- **配置場所**: docs/templates/example-feature/ ディレクトリ
- **更新頻度**: 機能の要件変更時
- **例**: [docs/templates/example-feature/PRD/requirements.md](/docs/templates/example-feature/PRD/requirements.md)

### 9. トラブルシューティングガイド

- **目的**: 一般的な問題とその解決方法を提供するドキュメント
- **対象読者**: エンドユーザー、サポート担当者、開発者
- **必須要素**:
  - 問題の症状と説明
  - 解決手順
  - 代替手段
  - 予防策
  - 関連するエラーコード
- **配置場所**: docs/guide/troubleshooting.md
- **更新頻度**: 新しい問題と解決策の発見時
- **例**: [docs/guide/troubleshooting.md](/docs/guide/troubleshooting.md)

### 10. 開発ロードマップ

- **目的**: プロジェクトの開発計画と今後の方向性を示すドキュメント
- **対象読者**: 全プロジェクトメンバー、ステークホルダー
- **必須要素**:
  - 短期・中期・長期の開発目標
  - 計画されている機能と優先順位
  - マイルストーンとタイムライン
  - リリース計画
- **配置場所**: docs/product/roadmap.md
- **更新頻度**: 四半期ごと、または大きな方向性の変更時
- **例**: [docs/product/roadmap.md](/docs/product/roadmap.md)

### 11. リリースノートとCHANGELOG

- **目的**: バージョンごとの変更点や新機能を記録するドキュメント
- **対象読者**: エンドユーザー、開発者、サポート担当者
- **必須要素**:
  - バージョン番号と日付
  - 新機能
  - 改善点
  - バグ修正
  - 非推奨・削除された機能
  - 移行手順（必要な場合）
- **配置場所**: docs/guide/releases/CHANGELOG.md
- **更新頻度**: リリース時
- **例**: [docs/guide/releases/CHANGELOG.md](/docs/guide/releases/CHANGELOG.md)

### 12. 開発フロードキュメント

- **目的**: 開発プロセス、ワークフロー、ドキュメント作成タイミングを説明するドキュメント
- **対象読者**: 開発者、プロジェクトマネージャー
- **必須要素**:
  - 開発ライフサイクルの各段階
  - 各段階での役割と責任
  - 必要なドキュメントと作成タイミング
  - レビューとフィードバックプロセス
- **配置場所**: docs/guide/developer/development-flow.md
- **更新頻度**: 開発プロセスの変更時
- **例**: [docs/guide/developer/development-flow.md](/docs/guide/developer/development-flow.md)

## ドキュメント種別の選択ガイドライン

新しいドキュメントを作成する際は、以下の手順で適切な種別を選択してください：

1. ドキュメントの主な目的を明確にする
2. 想定される主要な読者を特定する
3. 上記のドキュメント種別リストから最も適した種別を選択する
4. 選択した種別の必須要素と配置場所に従ってドキュメントを作成する
5. 必要に応じて、複数の種別の特性を組み合わせることも可能

## ドキュメント種別の拡張と変更

プロジェクトの進化に伴い、新しいドキュメント種別が必要になる場合や、既存の種別の定義を変更する必要が生じる場合があります。その際は、以下のプロセスに従ってください：

1. 新しい種別の目的、対象読者、必須要素を明確に定義する
2. チームメンバーとレビューし、フィードバックを収集する
3. 本ドキュメントを更新し、新しい種別を追加または既存の種別を変更する
4. 変更を全チームメンバーに周知する

## AIインストラクション

このルールを読んだAIは以下の行動をとってください：

- 定義された種別に従ってドキュメントを分類・作成し、種別の特性に合わせた構造と内容を提案する
- 既存ドキュメントの種別適合性を確認し、必要に応じて改善または適切な種別を提案する
- 不足している重要なドキュメント種別があれば、その作成を提案する
- プロジェクトに必要だが存在していないドキュメント種別があれば、その作成を提案する
- ドキュメント作成時に、選択した種別の必須要素が含まれているか確認し、不足している場合は追加を提案する
